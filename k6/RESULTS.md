## 2. Redisson 분산락 오버셀링 방지 검증 (POST /reservations, 1000 VUs, 재고 20개)

- 1000명 동시 예약 시도 → DB 기준 정확히 20건 성공, 0건 오버셀링
- 클라이언트(k6) 집계상 성공 19 / 재고부족 265 / 응답유실 716으로 나타났으나,
  DB의 remaining_quantity=0으로 실제 처리는 정확했음을 확인
  (응답 유실은 락 대기로 인한 서버 커넥션 포화가 원인으로 추정, 정합성 문제는 아님)
- 결론: 1주차에 재현한 오버셀링 문제가 3주차 분산락 적용으로 실제 부하 상황에서도 해결됨을 검증

## 3. 응답 유실 대비 - Idempotency Key 적용 및 재시도 시나리오 검증

- 부하테스트 중 발견한 문제: 서버는 예약을 정상 처리(재고 차감 완료)했지만 응답이 클라이언트에게 유실되는 케이스 발견
  (락 대기로 인한 서버 커넥션 포화가 원인으로 추정)
- 실무 위험: 응답을 못 받은 클라이언트(사용자의 재시도 또는 클라이언트 자동 재시도)가 동일 요청을 다시 보내면,
  같은 사용자가 같은 티켓을 중복 예약할 수 있음
- 해결: `POST /reservations`에 `Idempotency-Key` 헤더 지원 추가
  - Redisson의 `RBucket.trySet()`으로 최초 요청만 원자적으로 선점 (Map.putIfAbsent와 동일한 원리)
  - 처리 결과(예약 id)를 Redis에 저장해두고, 같은 키로 재요청 오면 DB 재처리 없이 기존 결과 반환
- 1차 검증 (수동): 동일 Idempotency-Key로 같은 예약 요청을 Postman으로 2번 전송
  → 첫 요청 201(신규 생성), 두 번째 요청 200(기존 결과 반환) → MySQL 확인 결과 재고는 1개만 정상 차감
- 2차 검증 (부하 상황 재현, reservation-idempotency-test.js): 응답이 애매하게 실패하면 동일 Idempotency-Key로
  최대 3회까지 자동 재시도하도록 시나리오 구성, 1000 VUs / 재고 20개로 재실행
  - k6 집계: 성공 20 / 재고부족 980 / 재시도 860회 발생 (응답 유실이 매우 많았음에도 재시도로 전부 명확한 응답 수신)
  - DB 확인: `remaining_quantity = 0`, `reservation` 테이블 실제 행 개수 = 20건 정확히 일치
  - → 860회의 재시도가 있었음에도 중복 예약 0건, Idempotency Key가 의도대로 동작함을 확인
- 참고(개선 여지): 재시도 시 지연 없이 즉시 재요청하도록 짜서 응답시간이 크게 늘어남(평균 9.3s, 최대 23.6s).
  실제 클라이언트라면 exponential backoff(재시도 간격을 점점 늘림) 적용이 필요 — 이번 프로젝트 범위에서는
  "정합성이 지켜지는가"까지만 검증하고, backoff 적용은 다음 개선 과제로 남겨둠
